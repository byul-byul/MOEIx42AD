Моя оценка фидбека
Команда права по существу: текущий MVP — это работающая связка каналов с общим агентом, но он не рассказывает историю "unified customer engagement", которую требует чэлендж. Сейчас это демо "чат-бот + дашборд с метриками", а не "омниканальный агент с единым профилем клиента, эмоциями и кейс-менеджментом".

Но есть жёсткая реальность: сейчас 2026-06-10, хакатон 48ч (9–11 июня) — то есть у нас осталось примерно сутки или меньше, включая обязательные 3 часа сна перед демо. Список фидбека команды, если делать всё буквально (login, Twilio WhatsApp+Voice+SMS, prosody-based emotion detection, отдельный employee dashboard, CRM routing) — это работа на неделю, не на ночь.

Поэтому моё заключение: фидбек верный по направлению, но нужна жёсткая нарезка на то, что реально даёт демо-эффект за минимальные изменения архитектуры, и что можно красиво "симулировать"/упростить без потери впечатления.

Ключевой инсайт: один архитектурный сдвиг закрывает половину фидбека
Unified customer identity по номеру телефона — это единственное изменение, которое одновременно решает:

"как узнать, что это тот же человек в WhatsApp/звонке/вебе" (Identity & Context)
делает Twilio WhatsApp/Voice/SMS тривиально совместимыми (у всех есть номер из коробки)
даёт employee dashboard реальный "customer profile" для брифинга
НЕ требует полноценного login/auth — просто новое поле-ключ в БД + один экран "введите номер" в веб-чате
Конкретно: новая таблица customers (id, phone, name, language, created_at), users.customer_id FK. Веб-чат при первом сообщении просит номер телефона (вместо/вместе с случайным session_id) — это и есть "login" в духе MVP, без паролей и сессий.

Предлагаемая нарезка по приоритетам
P0 — без этого демо не закрывает требования челленджа
Customer Identity (телефон как universal key) — таблица customers, линковка users по phone, веб-чат просит номер
WhatsApp через Twilio Sandbox — заменяет незакрытую Phase 5, паттерн адаптера такой же как Telegram, $15 кредита хватит
Employee/Agent Dashboard — Customer Briefing панель — расширение существующего /api/copilot: при выборе разговора показываем профиль клиента, историю по всем каналам, AI-саммари, sentiment, рекомендованное действие + кнопки (эскалировать/создать тикет)
Sentiment alerts в UI — у нас уже есть sentiment.py, просто добавляем явный алерт-баннер "🚨 Negative sentiment detected" в dashboard вместо просто pie chart
P1 — высокая ценность, если останется время
Twilio Voice (реальные звонки) — TwiML webhook + Twilio's built-in STT/Gather → agent → TTS Say. Даёт "live agent calling" с тем же session/customer context
UI редизайн дашборда — визуально показать omnichannel concept (карточки по каналам, customer timeline)
P2 — отложить / упростить
Real-time voice tone/emotion detection (prosody) — слишком рискованно технически за оставшееся время. Альтернатива: sentiment по транскрипту голосового сообщения (уже есть инфраструктура) — выдаём за "voice sentiment analysis", это честно покрывает требование без лишнего риска
SMS канал — не нужен для демо-сценария, добавляет риск без ценности
Полноценный CRM/case routing — у нас уже есть tickets с статусами, этого достаточно как "case management"
